# Workflow 流程设计 · 产出「阶段一开发文档」用

> 这份是「**怎么生那份文档的流程**」，不是阶段一文档本身。
> 目的：用代理大军，把阶段一开发文档做到 **95% 正确、可直接施工、且为之后搬去 SwiftUI 正式前端铺好路**。
> 现在只设计流程，等你点头才启动。

## 0. 范围
- 产出物：**`Workflow-1-看得到能用.md`**（阶段一开发文档，含前端＋后端＋衔接点）。
- 本流程只「写文档」，**不写程式**。

## 1. 核心原则 · 前端可替换（最重要）
现在云端做的是**过渡前端（Web）**；正式前端是之后 **Xcode + SwiftUI**。为了搬迁时工作量最小，文档强制两件事：

1. **UI 层 / 服务层 分离**
   - 画面（UI）只管长相与互动，**绝不直接呼叫后端**。
   - 所有后端呼叫，统一走一层「**服务层 / API 客户端**」。
   - 搬去 SwiftUI 时：**重画 UI + 照同一份契约重做服务层**，后端＋API 一字不改。

2. **所有 API 触点登记表**
   - 文档里有一张「触点登记表」：前端每一个会碰后端的地方，列出
     `触点名称 → 对应 API 端点 → 输入 → 输出 → 权限 → 之后 SwiftUI 怎么接`。
   - 这张表就是「**搬迁清单**」，照着接就好。

## 2. 代理大军编制（约 23 个）
| 连队 | 人数 | 角色与工作 |
|---|---|---|
| 指挥 | 1 | 总指挥（我）：拆任务、合并、把关 95% |
| **侦察连** | 8 | 只查不写，各交一页「事实＋最佳做法＋雷区」（题目见下） |
| **工兵连·后端** | 4 | ①资料表&RLS ②Auth&删除帐号 ③离线同步 ④API 契约 |
| **工兵连·前端** | 4 | ⑤设计系统 ⑥5 个画面规格 ⑦旋钮&双模式空壳 ⑧服务层&**触点登记表** |
| 工兵·验收 | 1 | Step 0 验收页规格＋每盏灯的验收标准 |
| **搬迁兵** | 1 | SwiftUI 映射：每个 Web 触点 → 之后 SwiftUI 怎么接 |
| **红队（驳斥）** | 3 | 完整性 / 正确性 / 搬迁&上架风险，逐章攻击 |
| 整合兵 | 1 | 合成一份、对齐触点表、检查零矛盾 |

侦察连 8 个题目：
1. Supabase Auth + Sign in with Apple（iOS 流程、token、帐号删除 API）。
2. UI/服务层分离在 Web 前端怎么落地，且方便之后映射 SwiftUI。
3. Postgres 资料表 + RLS「只能看自己资料」标准写法。
4. 离线先存、有网再同步的最简可靠做法（updated_at / 软删除 / 冲突）。
5. 过渡前端该用什么（纯 HTML/JS vs 轻框架），最好结构贴近 SwiftUI、好丢弃。
6. SwiftUI 映射：Web 的画面/元件/服务层 → SwiftUI 的 View/ViewModel/Service。
7. API 契约惯例：端点设计、错误格式、Auth header（PostgREST vs 自建端点）。
8. 阶段一相关的上架硬需求（删帐号 UI、隐私基本、Sign in with Apple）。

## 3. Workflow 七阶段（流水线）
```
0 准备：载入蓝图 01/02/03 当唯一真相
1 侦察：4 侦察兵各交一页「事实＋最佳做法＋雷区」
2 起草：4 工兵各写自己的章节（继承蓝图契约，不重新发明）
3 驳斥：2 红队逐章攻击，列缺陷清单
4 修补：工兵照缺陷修
5 整合：整合兵合成一份，对齐「触点登记表」前后一致
6 评分：照 95% 检核表打分；不到就回 3，循环到 ≥95%
7 产出：交付 Workflow-1 文档 ＋ 触点登记表附录
```

## 4. 阶段一文档的目录（产出长这样）
1. 范围与目标
2. 系统架构图（前端 / 服务层 / 后端 三层，标明前端可替换）
3. 后端：资料表、RLS、Auth 流程、同步策略
4. **API 契约**：每个端点（路径、输入、输出、权限、错误）
5. ★ **API 触点登记表**（搬迁清单）★
6. 前端：可替换资料夹结构（UI 层 / 服务层）、设计系统、5 个画面规格、旋钮＋双模式空壳
7. Step 0 验收页规格
8. 验收标准（每盏灯怎样算亮）
9. 代理执行计划（这阶段开发时怎么派工）
10. 95% 检核表（勾完才算过）

## 5. 95% 检核表（评分标准，节录）
- [ ] 前端没有任何一处直接呼叫后端（全走服务层）
- [ ] 每个画面用到的后端能力，都在触点登记表里有一条
- [ ] 每个 API 端点：输入/输出/权限/错误都写清楚
- [ ] 资料表 + RLS 完整，且「只能看自己资料」
- [ ] Auth 流程含「删除帐号」
- [ ] 每个步骤都有可验收的「灯亮条件」
- [ ] 搬去 SwiftUI 的步骤明确、工作量被压到最小
- [ ] 与蓝图 01/02/03 无矛盾

## 6. 给每个工兵的两张纸
- **设计指令（做什么）**：从蓝图来——功能、契约、限制；前端必须 UI/服务层分离；金钥不进前端。
- **执行指令（怎么做＋怎么验）**：继承既有契约不重造；产出要能过 95% 检核表；每条 API 触点都要可被 SwiftUI 重接；写完自检再交整合兵。
