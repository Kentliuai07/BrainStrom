# ① 基础技术选型文档

> 经 12 个探勘代理「正反辩论」后收敛的技术栈。每一项都附「为什么选 + 最强反方 + 结论」。
> 目标：iOS App，上架 App Store。v1 范围 = 做到「市集之前」（个人 AI 笔记 + 全局 AI + Git 进度）。

## 关键「相互驳斥」与结论（代理辩论摘要）

1. **SwiftData(iOS17+) vs Core Data(iOS13+)**
   反方：iOS 17+ 会少掉旧机用户。
   结论：**用 SwiftData/iOS 17+**；除非上架后数据显示旧机占比高，再考虑兼容。

2. **Supabase vs Firebase**
   反方：Firebase 离线/即时更成熟。
   结论：**Supabase**——内建向量、成本可预测（固定月费 vs 按读取计费贵 3–5 倍）、开源不锁死。我们不是重协作 App。

3. **pgvector vs 专用向量库（Pinecone…）**
   反方：百万笔以上专用库召回更好。
   结论：**pgvector**；我们 50 笔等级绰绰有余。若全局搜寻召回率 <80% 再换。

4. **结构化：tool_use+重试 vs 纯 JSON 模式**
   反方：纯 JSON 一次到位、更快更省。
   结论：**tool_use + 校验 + 重试**；幻觉是致命伤，绝不裸用 JSON 模式。Sonnet 成功率 <90% 时升 Opus。

5. **离线：后写覆盖 vs CRDT**
   反方：同一块在两装置同时改，会悄悄丢一边。
   结论：**先后写覆盖**；单人低冲突。真出现热点冲突，再上 Yjs（1–2 天）。

6. **AI 算力：Supabase Edge Functions vs Fly.io（你的实测重点）**
   反方：全部塞 Supabase 最省事、零维运。
   结论：**AI 代理独立放 Fly.io**。你实测过「用 Supabase 当 LLM 云端」会慢（冷启动＋区域＋串流限制）。Fly.io 常驻机器不冷启动、可选贴近 Claude 机房的区域、长串流更稳；资料/登入/向量仍留 Supabase。代价：多一个平台要顾（月费仅几美元）。

7. **语音转写**
   结论：**延后到 v2 之后**，v1 不做（你指定）。届时再比 Whisper vs 端上。

8. **CI：Xcode Cloud vs GitHub Actions**
   反方：Xcode Cloud 原生整合更紧。
   结论：**GitHub Actions + Fastlane**；省席次费、脚本进仓库不锁死。
