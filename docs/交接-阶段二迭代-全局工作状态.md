## 2. 现在的项目前后因果（怎么走到这一步）

1. **产品身份**：BrainStrom 是「AI 开发笔记 App」。架构 = iPhone(SwiftUI) + Supabase(Postgres+pgvector，资料真相) + Fly.io(常驻 AI 代理，锁 Claude 金钥)。现在是**过渡期 Web 前端 + 模拟后端**（localStorage），真后端还没建。
2. **三阶段**：一「能写」(Step 0–2，地基+笔记，无 AI，骨架已有) → 二「变聪明」(Step 3–7，AI 功能) → 三「能上架」(Step 8–9，付费+私密；Step 10 市集延后 v2)。
3. **起点需求**：AI 把乱写笔记结构化没问题；但**第二次改了原文不要整篇重做**（省钱 + 不洗掉已整理/手改内容）。使用者说 SBIR_NEW 有现成模组。
4. **研究 SBIR_NEW**：派代理大军扫，**忠实记录**了它的增量模组。关键发现：它靠「使用者点名段落号」而非自动 diff；保护靠「不送 AI」而非锁；RAG 没真做。→ 决定**参考重写**，不搬码。
5. **规划阶段二**：发现阶段二**根本还没有详细文档**（只有阶段一 Workflow-1）。→ 决定先写阶段二文档。
6. **开发顺序拍板（方案 A）**：派代理中立判断「AI 功能能不能整套先做」→ 结论：不行，AI 功能站在地基上（登入/笔记/blocks），且彼此有依赖。→ 顺序定为：**先地基+共用引擎 → 用最简单的「单专案聊天」验证引擎 → 结构化 → 加模组 → 增量 → 全局 → GitHub**。
7. **几个细节决策**：
   - 卡片**先不固定版型**：只教 AI「每个主题一张卡」，不限种类/样式/字数（20 卡清单只当参考）。
   - 加模组 **UI 简化成横排按钮**（不做花俏圆形旋钮），但加卡功能照做。
   - GitHub 有**两个**功能：①判断进度（拿 buildSteps 对比 commit）②找呼应的开源项目。
8. **写出阶段二文档**：`docs/阶段二开发文档-变聪明-AI功能.md`，6 探勘 + 1 Plan + 3 核实代理验证，修掉 9 处与代码不符，到 **98%+**。
9. **★最新共识（本轮的转折，尚未写进文档，新窗口要据此迭代）★**：使用者重新想了「结构化散文 + 手改保护」这块，结论改成——
   - **原始散文（手写乱稿）：AI 不能动**，永远保留当原稿。
   - 结构化时，从**同一份结构化内容**渲染出**两种皮**：「优化散文」（排版优化、归类、按卡片顺序排成一篇顺的文章）+「卡片」。
   - **AI 可以改「优化散文」和「卡片」两边**（**推翻**之前「手改卡永不碰」的旧铁律）。
   - 「优化散文」和「卡片」**两边同步**。
   - **安全网从「锁定」改成「版本快照 + 一键还原」**（因为 AI 现在能动手改）。
   - **省钱机制保留**：原始散文没变（指纹比对）→ 不叫 AI；变了→只重优化变动的部分。
   - **架构建议（关键）**：「优化散文」= 卡片的另一种呈现，**不要存两份**（底层只有一份有顺序的卡，串起来就是散文，拆开就是卡）→ 同步天生免费、不会两边打架。

## 3. 现在开发的系统功能全景（阶段二 · 方案 A 顺序 · 含最新三层内容模型）

### 3.1 资料的「三层内容模型」（最新共识，迭代的核心）

- **第 1 层 · 原始散文**：使用者手写的乱稿。**AI 永不改写**；使用者本人仍可继续在这里写（它是结构化的输入来源，也是「原文变了要重整」的触发点）。是安全网原稿。
- **第 2 层 · 一份结构化内容**：AI 从原始散文优化出来的「一串有顺序的卡」。这是**唯一一份**结构化资料。
- **两种皮（同一份资料的两种呈现）**：
  - **优化散文**：把那串卡按顺序串起来、排版顺、归类好，呈现成一篇文章。
  - **卡片**：把那串卡拆开一张张显示。
- AI 可改这一份结构化内容（不管你从散文皮还是卡片皮去改，改的都是同一份）；改前存版本，可一键还原。原始散文不受影响。

### 3.2 七步功能（每步：目标 + 依赖）

- **前置 · 资料库补栏**：blocks 加 `source/locked(改语意,见下)/sourceAnchor/structureGen`；systems 加 `sourceHash/ai_restructure_count/structuredAt`；新表 embeddings/chat/repo。**注意 `locked` 在新模型里语意要重想**（旧模型是「永不碰」，新模型 AI 可碰，所以 `locked` 可能改成「使用者偏好/钉选」之类，或拿掉——这是要新窗口重新设计的点）。
- **Step 1 · AI 共用引擎**：Fly.io 常驻代理 + 统一入口（SSE 事件协议 / gateway 三入口 / parse-ai-json 5 层 / 成本阀 / AuditContext）。金钥不进前端。
- **Step 2 · 单专案聊天**：最简单 AI，验证引擎。AI 知道该专案全部内容（含手动加的卡）。
- **Step 3 · AI 结构化（第一次）**：原始散文 → 产出「一份结构化内容」→ 同时能看成优化散文 + 卡片。
- **Step 4 · 加模组**：横排按钮加卡；手动加/改的卡做标记（新模型里标记语意要重想）。
- **Step 5 · 增量结构化（第二次）**：原文指纹没变不叫 AI（省钱）；变了自动 diff、只重优化变动部分；AI 可动既有内容；版本快照 + 一键还原当安全网。
- **Step 6 · 全局 AI 跨笔记找回**：摘要+向量(embeddings)，pgvector 搜寻，指出在哪则，free 上限 50。生向量/写向量/查向量三步要自己写（SBIR 没做）。
- **Step 7 · 连 GitHub**：①进度（buildSteps 对比 commit，环形%+打勾）②找呼应的开源项目（拿项目内容去 GitHub 搜相似 repo 推荐）。

### 3.3 前端验收点（使用者在前端验收，挂 `acceptance.html` 灯）

阶段一 6 盏 + 阶段二 6 盏（`ai_engine/chat_note/structure/structure_incremental/global_recall/git_progress`）。核心验收动作：
- (a) AI 流式响应：聊天送一句 → 回答逐字蹦出 + token 用量 + 错误显示。
- (b) 单篇内容识别：加一张含关键字的卡 → 问 AI「这则在讲什么」→ 看它有没有复述出来。
- (c) 结构化逐张浮现 / 增量「改了原文只动变动处、原文没变不花钱、改坏能还原」/ 全局找回命中正确笔记 / GitHub 进度环形图。
> **新模型注意**：验收点要补「优化散文 ↔ 卡片 两边同步」「原始散文没被动」「版本还原」这几个新动作。

### 3.4 可复用的 SBIR_NEW 模式（照抄，省力）

SSE 事件协议（delta/done/error/usage/credit_update + 自取 card_start/card_done）；单专案聊天三层 system block + `cache_control:ephemeral` 快取；不用 tool_use 而用「prompt 写死 JSON + parse-ai-json 5 层兜底」（阶段二文档建议改用 tool_use 为主，待定）；两层重试（gateway maxRetries=2 实退避 1s→2s；spec-generator 那条 stream 3 次 2s→4s→8s）；成本阀（model 白名单 + max_tokens 8192 + 每日上限 shadow/canary/full + 限流）；摘要 fallback 取前 400 字。
