## 1. 一句话与设计哲学

**一句话：** 一个专案不再只是「一篇笔记」，而是一个有「身份」、能「自由记录」、还有「专属 AI 教练」的完整工作台。

**设计哲学（三句话记住）：**

1. **「一个专案 = 一张身份证 + 一叠笔记 + 一位教练」。** 身份证是这个专案的硬规格（用什么技术、放哪台服务器），笔记是你随手乱写的草稿，教练是看得到全貌、帮你想清楚的 AI。
2. **「该乱的地方让你尽情乱，该整齐的地方绝对整齐」。** 开发笔记可以随便写、贴连结、塞模组，怎么乱都行；但系统结构（身份证）必须干净、准确、可信任。
3. **「AI 是唯一的守门员，但永远只建议、不强迫」。** 任何东西要进系统结构，都得先过 AI 这一关（无后门）；但 AI 只会「建议」，最后一定是你点确认才算数。

这一阶段不是加功能，是**把房子的格局重新打通**：从「单层一个笔记页」升级成「专案首页（三分页）→ 多笔记内页」的四层导航树。

## 2. 三分页架构：AI 教练 / 开发笔记 / 系统结构

进入任何一个专案，顶部是三个分页（沿用既有 `segButton` 工业风格选项卡）：

| 分页 | 图示 | 白话职责 | 谁能写 |
| --- | --- | --- | --- |
| **AI 教练** | 🤖 | 看得到「整个专案」（身份证＋所有笔记），陪你发想、帮你想清楚 | AI 对话，人提问 |
| **开发笔记** | 📝 | 你的草稿本：随便写、贴 YT/GitHub 连结、加模组、请 AI 整理 | 人写为主，AI 协助 |
| **系统结构** | 🪪 | 专案的「身份证」：硬规格（名称 / 技术栈 / 资料库 / 上线方式） | **只有 AI 能写，人只能看** |

### 各自职责讲清楚

**🪪 系统结构（身份证）**
- 字段（硬需求）：**名称 / 技术栈（前端 · 后端 · API）/ 资料库 / 上线（服务器 · 部署方法）**，选填可扩充。
- **铁律：人只能看，不能直接改。** 要改？去跟 AI 讲，AI 给建议，你点确认。
- 空字段显示淡色占位：「— 尚未填写，编辑或让 AI 记录」。

**📝 开发笔记（智能笔记）**
- 一个专案可以有**很多篇**笔记（这一阶段才正式打开多笔记）。
- 预设「全看」：AI 默认会读整篇笔记。
- 少数内容可加 🔒 排除标记，AI 就看不到那一块（隐私 / 不想被 AI 干扰的草稿）。
- 可以：写字、贴连结（YT / GitHub）、加模组、按钮请 AI「记进结构」。
- 沿用既有「文章 / 卡片」双视图、Markdown 渲染、⚡点子助攻、版本链 undo/redo。

**🤖 AI 教练**
- 每个专案**一位**教练，看得到全貌（身份证＋所有笔记切片）。
- 两种玩法（第 6 节详述）：**引导发想（扩散）** 和 **结构厘清（收敛）**。
- 教练**绝不直接改任何东西**，只透过 proposal 建议钮给你选。

## 3. AI 守门员铁律：任何内容进结构，一律过 AI

**这是阶段三最重要的一条规矩，写死、不许有例外。**

> 任何东西（一个模组、一个随手灵感、一条贴上来的连结、任何内容）想进入「系统结构（身份证）」，
> **一律先经过 AI**（没有后门、没有手动直填的偷吃步），
> AI 产生建议 → **使用者点确认** → 才真正写入身份证。

### 资讯流图（文字版）

```
[你做的任何动作]
  · 在笔记里写「前端用 React」
  · 贴一条 Fly.io 部署连结
  · 拖一个模组进来
  · 在 AI 教练里聊到「资料库用 Postgres」
        │
        ▼
┌─────────────────────────────┐
│   AI（唯一守门员，无后门）      │  ← 后端 prompt 侦测「硬需求」
│   侦测到这是身份证该记的东西？    │
└─────────────────────────────┘
        │ 是
        ▼
   AI 吐出 proposal 建议钮
   「📇 记入结构：前端 = React」
        │
        ▼
   ┌──────────────┐
   │  使用者点确认？  │
   └──────────────┘
     │是          │否
     ▼            ▼
  写入身份证     什么都不发生
  (update_spec)  (结构保持原样)
```

**重点白话：**
- 系统结构页**没有「+ 新增」「直接编辑」按钮直通写入**。所有写入路径都收敛到 `update_spec` 这条「过 AI → 点确认」的通道。
- AI 只是**建议者**，不是决定者。你不点，永远不会变。
- 这样保证身份证永远「干净、可信、有人把关」，不会被随手乱填污染。

## 4. 资料模型变更

阶段三动三件事：**打开多笔记**、**新增身份证 SystemSpec**、**笔记智能排除 🔒**。底层其实大多已备妥，改动集中、可控。

### 4.1 打开多笔记（底层早就支援，只差 UI 入口）

- 现状：`SystemEntity.notes: [NoteEntity]` 一对多**早已存在**，`NoteEntity` 含 `systemID` 外键、级联删除（`deleteRule: .cascade`）。
- 但目前 UI 只用 `documentNote()` 惰性取「最新单笔」，等于假装只有一篇。
- **改动：** 不动资料模型，新增 `notes(in: systemID)` 查询（仓储已可提供），UI 层新增「笔记清单」入口。`NoteDocument` 初始化要支援传入 `noteID`，不再假设系统内笔记唯一。

### 4.2 新增系统身份证 SystemSpec

**领域模型（DomainModels.swift）新增 struct：**

`SystemSpec`：`name?` / `frontend?` / `backend?` / `apis: [String]?` / `database?` / `server?` / `deployMethod?` —— 全部 optional String，选填、可扩充。

**持久化（二选一，建议方案 A 内嵌 JSON，最省事、仿既有 `tagsData`）：**
- **方案 A（推荐）：** `SystemEntity` 新增 `systemSpecData: Data?`，JSON 编码 `SystemSpec` 存进去。改动最小，不新增 entity，不动关联。
- 方案 B：新建独立 `SystemSpecEntity`，一对一关联 `SystemEntity`。结构更正，但改动较大，留作日后。

**仓储新增（NotesRepository + 协议 NotesRepositoring）：**
- `func systemSpec(systemID: UUID) throws -> SystemSpec?`
- `func updateSystemSpec(systemID: UUID, spec: SystemSpec) throws`

**传给 AI 的上下文（AIServicing.swift）：**
- `NotePayload` 新增可选字段 `systemSpec: SystemSpec?`，让 AI 知道现有技术栈，避免重复建议。

### 4.3 笔记智能排除 🔒

- 现状：Block 层级已有 `source`（manual / ai / notes）和 `deletedAt`（软删）；Note 层级有 `lastAiHash`（hash gate）。**缺**「这块要不要给 AI 看」的标记。
- **改动：** Block 层级新增 `excludedFromAI: Bool`（预设 false = 全看）。组 payload 给 AI 时，`excludedFromAI == true` 的 block 直接剔除，不送出。
- UI：该 block 加一个 🔒 小钮切换排除状态。

### 4.4 改动清单一览（约 7 个档案）

| 档案 | 改什么 |
| --- | --- |
| `DomainModels.swift` | 新增 `SystemSpec` struct；`Nudge` 扩充 `specSnapshot?`（沿用快照模式） |
| `AIServicing.swift` | `NotePayload` 加 `systemSpec?`；proposal action 加 `update_spec` |
| `PersistentModels.swift` | `SystemEntity` 加 `systemSpecData: Data?`；Block 加 `excludedFromAI: Bool` |
| `NotesRepository.swift` | 加 `systemSpec(_:)` / `updateSystemSpec(_:_:)` / `notes(in:)` |
| `NoteDocument.swift` | 加 `systemSpec` 属性 + `updateSystemSpec(_:)`（仿 `updateNudge` 写盘语义）；payload 注入 spec、剔除 🔒 block |
| `NoteChatViews.swift` | `proposalRow()` 的 switch 加 `case "update_spec"`：解析 JSON → 写入 → 标记 used |
| 新档 `SystemStructureView.swift` | 身份证只读 / 展示页 |

## 5. 导航变更：从「单层一页」到「四层树」

### 现状（问题）

- 首页 `NavigationStack(path:)` 直接用 `UUID` 开 `NoteScreen`，`HomeRoute` 枚举只有 `.settings`。
- 「文章 / 卡片」是 `NoteScreen` 内部的 `ViewMode` 状态切换，不是页面导航。
- 等于：点专案卡 → 直接进一个单笔记编辑页。**没有专案首页，也没有身份证、没有教练。**

### 目标（四层导航树）

```
第 1 层  专案列表（首页 HomeScreen）
            │ 点专案卡
            ▼
第 2 层  专案首页 SystemDetailScreen  ← 顶部三分页
            ├── 🤖 AI 教练  AICoachView
            ├── 📝 开发笔记  NotesListView（多笔记清单）
            │       │ 点某一篇
            │       ▼
            │   第 3 层  单篇笔记内页 NoteDetailScreen（文章/卡片在此）
            └── 🪪 系统结构  SystemStructureView（身份证）
```

### 具体改动

- **新增 `HomeRoute.systemDetail(id: UUID)`** 关联值路由；`HomeRoute` 成为**唯一路由源**，不再混用 `UUID` 直导。
- 首页专案卡点击：`path.append(HomeRoute.systemDetail(id: system.id))`。
- **新建 `SystemDetailScreen.swift`**：`@State selectedTab` 驱动三分页（`.coach` / `.notes` / `.structure`），用 `systemTabBar()` 组件（沿用 `NoteScreen` 既有 `segButton` 样式）。
- **新建 `NotesListView.swift`**：查询同 `systemID` 的笔记清单，点选进入单笔记。
- **新建 `NoteDetailScreen.swift`**：由现有 `NoteScreen` 改造，**保留全部 UI**（navBar / dock / fab / dialLayer / 文章/卡片），只移除「systemID 唯一」假设、改 `dismiss` 目标。
- **新建 `AICoachView.swift`**：读身份证＋当前/全部笔记切片，驱动教练对话。
- **新建 `SystemStructureView.swift`**：身份证只读展示。

> 既有 `ArticleView` / `CardsView` / `BlockRow` / `CardRow` **无需修改**，整组下沉到 `NoteDetailScreen` 当内容区即可。

## 6. AI 教练：两种玩法 + 记入结构

教练沿用既有串流（SSE）+ proposal 建议钮架构，**80% 基础设施可直接复用**（`ChatViewModel.runStream`、`Nudge` 快照、`MarkdownView`）。后端靠一个 `kickoff` 旗标与系统提示词切换两种模式。

### 6.1 引导发想（扩散模式）

**白话：** 帮你「把脑袋打开」，问开放问题，丢出一堆小字卡让你挑、让你联想，先别急着收敛。

触发：点 ⚡ 点子助攻 / 教练开场（`kickoff = true`）。

教练会围绕这几个泡泡发问（小字卡清单）：

- 🎯 **痛点**：你到底想解决什么问题？
- 📈 **市场**：谁会用？目标族群是谁？
- 🥊 **竞品**：市面上有人做了吗？他们怎么做？
- 🎬 **场景**：使用者典型的一天是怎样用到它？
- ✨ **核心功能**：先上线的 3 个功能是什么？

结尾必带 1–4 个 proposal 钮，例如：`['📇 记入结构', '✦ 优化笔记', '▦ 整理成卡片']`。

### 6.2 结构厘清（收敛模式）

**白话：** 帮你「把想法收拢成规格」，一格一格把身份证填好。

触发：发想之后，或直接进入（`kickoff = false`）。两种呈现：

- **卡片式**：教练把目前抓到的规格做成卡片让你确认 / 补充。
- **问答式**：哪个字段含糊，就追问一句把它问清楚。

当你提到硬需求（如「前端用 React」「部署在 Fly.io」），AI 侦测到 → 吐 `update_spec` proposal。

### 6.3 update_spec：把建议记入结构

- proposal 格式：`{ action: "update_spec", label: "记入结构：前端=React", instruction: <JSON 序列化的 spec 片段> }`。
- 前端：`NoteChatViews.proposalRow()` 的 switch 加 `case "update_spec"` → 解析 instruction 里的 JSON → 呼叫 `NoteDocument.updateSystemSpec()` → 标记 `proposalsUsed`（整列禁用，防重复）。
- 后端（Fly.io）：系统提示词加「硬需求侦测」逻辑，侦测到就穿插 proposal 事件。SSE 协议**不用改**，proposal 事件早已存在，只是多一个 action 值。
- **呼应第 3 节铁律：** 这是唯一写入身份证的通道，且必须「使用者点钮」才生效。

## 7. 省 Token 策略

目标：**AI 只在被叫时读 + 只送当前专案切片 + prompt 快取 + hash/diff 闸**，整体省 45–90% 成本。全部可在现有框架内落地，不用大重构。

### 7.1 只送切片（不要把整个专案塞进去）

- 三层上下文组织：
  - **L1 身份证**（小，约 500–1k token）：稳定，挂长快取。
  - **L2 当前笔记**（变动部分，2–10k）：编辑频繁，挂短快取。
  - **L3 跨笔记**（仅在需要时才捞，且只送摘要）：不快取。
- 🔒 排除的 block 一律不送（第 4.3 节）。
- pinned / 模组 block 不进 AI 输入。

### 7.2 Prompt 快取（Claude 官方做法）

> 依 Claude 官方文件：在 system / 内容区块上标 `cache_control: { type: "ephemeral" }`，命中快取的 `cache_read` token 只算约 0.1x 价钱，省约 90%。

- **L1 身份证**：规格稳定 → 用较长 TTL（如 1h），搭配 `lastSpecHash` 侦测变动。
- **L2 当前笔记**：编辑频繁 → 用短 TTL（5min）。
- **L3 跨笔记摘要**：结果多变 → 不快取。
- SSE `usage` 事件扩充：`{ input_tokens, output_tokens, cache_read_input_tokens, cache_creation_input_tokens, model }`，前端可显示「本次省费 XX%」badge（`savings = (1 - actualCost/baselineCost) × 100`）。
- 实作位置：Fly.io 后端改用 Anthropic SDK，在请求体加 `cache_control`。前端 SSE 解析架构**完全相容**，只多接 usage 字段。

### 7.3 hash / diff 闸（条件命中直返，零成本）

- **整篇 hash 闸**：算当前内容 hash，跟 `lastAiHash` 比对，一样 → 直接返回上次结果 + toast「无变更，零成本」，根本不呼叫 AI。
- **块级 diff**：每个 block 带 `aiHash`，只送「新增 / 变动」的 block（标 `changed: true`），避免「影印机效应」（重复重写越漂越远）。
- **身份证 hash 闸**：新增 `lastSpecHash`，身份证没变就 skip 重新询问。
- 这套 `aiHash` + `lastAiHash` 机制**现成已实装**，只需补 `lastSpecHash`。

## 8. 连结 / YT 读取（排在最后面做）

**为什么排后面：** 「侦测连结」很简单（前端正则即可），但「读懂连结内容」需要后端工具配合，依赖度高，所以放最后。

**做法（沿用 Claude 官方 tool use）：**

- 三个后端工具（locked，全在 Fly.io，**绝不在 iOS app 里跑**）：
  - `find_youtube`：收 YT URL → YouTube API（或 Claude `web_fetch`）抓标题 / 时长 / 字幕 → AI 读字幕做摘要。
  - `find_github`：收 repo URL → GitHub API（OAuth token 存 Fly.io）抓 stars / 语言 / README。
  - `find_info`：收一般网址 → Claude `web_fetch`（server tool，Anthropic 端执行）抓 + 解析 + 摘要。
- 这些都用 Claude `tool_use` + JSON schema 受控输出。
- 新端点：`POST /media/fetch`（收 URL + media_type 提示）→ 回结构化 metadata + 摘要给 iOS 显示。
- proposal action 已含 `find_youtube` / `find_github` / `find_info`，前端钮**早就准备好**，等后端工具上线即可接。

**阶段三内的优先序：** 先做「侦测 + 显示连结预览占位」，真正的「读懂内容」放到这一阶段最后一刀，做不完可顺延，不挡前面验收。

## 10. 实作步骤拆分（切 4 刀，每刀可上 TestFlight）

原则：**每一刀都能独立编译、独立验收、独立上 TestFlight**，做不完不互卡。

### 🔪 第 1 刀 · 导航骨架 + 三分页空壳（地基）

- 加 `HomeRoute.systemDetail(id:)`，首页改路由。
- 新建 `SystemDetailScreen` + `systemTabBar()` 三分页。
- 三页先放空壳：教练页占位、笔记页先塞「目前单笔」、结构页占位。
- **验收：** 点专案卡 → 进专案首页，三分页可切换，旧笔记照常能编辑。
- **风险：** 路由从 UUID 改 HomeRoute 可能影响既有返回流程 → 低，集中在 `HomeScreen`。

### 🔪 第 2 刀 · 系统结构（身份证）只读 + update_spec 通道

### 12.2 洞2 · update_spec（「记入身份证」钮）前后端契约（第2刀）

#### 核心约束（决定一切）
SSEParser 把 proposal item 的 `args.instruction` 当 **String** 取出，塞进 `ProposalItem.instruction: String?`。因此 **update_spec 的结构化资料必须 stringify 成一段 JSON 字串**，放在 `args.instruction` 里。**不改 SSEParser、不改 ProposalItem 型别**——零侵入复用既有 proposal 管线。

#### SSE proposal 事件原文（后端 → App，wire 格式）
```
data: {"type":"proposal","items":[{"action":"update_spec","label":"📇 记入身份证","args":{"instruction":"{\"name\":null,\"frontend\":\"React\",\"backend\":null,\"apis\":[\"Stripe\"],\"database\":\"PostgreSQL\",\"server\":\"Fly.io\",\"deployMethod\":null}"}}]}
```
- 整笔事件是 **一行 `data:` + 空行**（逐行扫，空行 flush）。
- `args.instruction` 的值是 **被转义的 JSON 字串**（里面的 `"` 要 `\"`），不是 nested object——wire 上的硬约束。
- `label` ≤12 字，建议固定前缀 `📇 记入身份证`，可按字段定制如 `📇 记入:前端=React`。

#### instruction 内 JSON（解开转义后）= SpecPatch（部分更新补丁）
字段对齐设计文档第一版（name / frontend / backend / apis[] / database / server / deployMethod，全选填）。规则写死：
- **null 或缺键 = 此次不动该字段**（保留旧值）；**非 null = 覆盖该字段**。语义是**部分更新（merge），不是整包覆盖**。
- `apis` 给了就**整阵列替换**（非逐项 append）；后端若只想加一个 API，应把「合并后的完整 apis 阵列」一起给。
- 后端不得发空补丁（7 字段全 null）；至少 1 个非 null 才发钮。

#### SystemSpec 型别（前端新建，目前 0 存在）
```swift
struct SystemSpec: Equatable, Codable, Sendable {
    var name: String?
    var frontend: String?
    var backend: String?
    var apis: [String] = []
    var database: String?
    var server: String?
    var deployMethod: String?
}
```
持久化挂在「系统」层级（SwiftData，与自由笔记并存），非塞进 Note.blocks。

#### 前端解析→写入→标记 used（写死流程）
在 NoteChatViews.swift proposalRow 的 switch 增加 case，**update_spec 不可加进 locked 集合**（否则被当「即將推出」拦掉）：
```swift
case "update_spec":
    noteVM?.applySpecPatch(doc, instructionJSON: item.instruction)
```
ViewModel 端解析与合并（null/缺键不动）：
```swift
func applySpecPatch(_ doc: NoteDocument, instructionJSON: String?) {
    guard let json = instructionJSON?.data(using: .utf8),
          let p = try? JSONDecoder().decode(SpecPatch.self, from: json) else {
        toast.show(String(localized: "身份证资料读取失败")); return
    }
    var spec = doc.systemSpec
    if let v = p.name        { spec.name = v }
    if let v = p.frontend    { spec.frontend = v }
    if let v = p.backend     { spec.backend = v }
    if let v = p.apis        { spec.apis = v }      // 整阵列替换
    if let v = p.database    { spec.database = v }
    if let v = p.server      { spec.server = v }
    if let v = p.deployMethod{ spec.deployMethod = v }
    doc.updateSystemSpec(spec)                       // 持久化
    toast.show(String(localized: "已记入身份证"))
}
private struct SpecPatch: Decodable {
    var name, frontend, backend, database, server, deployMethod: String?
    var apis: [String]?
}
```
**标记 used 免写**：proposalRow 进 switch 前已呼叫 `markProposalsUsed(bubble.id)`，整列立即变灰 disabled，天然防重复记入。

#### 后端发钮时机 + 注意
- chatNote/optimize/structure 任一流中，Claude 侦测到使用者陈述硬需求（技术栈/资料库/部署）→ 在 done 之前发一笔 proposal 事件。
- **第一版后端一次只发 1 个 update_spec item**（见下方 openRisk：整列禁用与「批量个别取消」冲突）。
- update_spec 是**唯一 instruction 内放 JSON 的 action**（edit_text 的 instruction 是纯文字），前端解析务必 `try?` + 失败给 toast 不崩。

**⚠️ 待确认（前后端拍板）**：
1. 「一批建议可个别取消」与现有「整列禁用」冲突 → 第一版只发 1 个 item；要批量需把 `proposalsUsed: Bool` 改 `Set<Int>`（额外前端工）。
2. proposalRow 只渲染前 4 个 item，update_spec 要排阵列前面别被截掉。
3. AI 反向读身份证当上下文 → 要给 optimize/structure/chatNote 三个 Body 加 spec 字段（后端线）。

### 12.4 洞4 · 教练多笔记切片 + token 预算（第4刀，补强第 7 节）

#### ProjectContext 结构（新增，放 AIServicing.swift）
```swift
/// 教练专用:整个专案的三层上下文切片。单笔记 optimize/structure 不用这个,继续用 NotePayload。
struct ProjectContext: Equatable, Codable, Sendable {
    let spec: SystemSpec?            // L1 身份证(稳定,挂长快取),nil 时整段省略
    let currentNote: NotePayload     // L2 当前笔记全文(挂短快取),复用既有 NotePayload
    let otherNotes: [NoteDigest]     // L3 其他笔记短摘要(不快取),已做预算裁剪
    struct NoteDigest: Equatable, Codable, Sendable {
        let id: String; let title: String; let summary: String
        let updatedAt: Date; let truncated: Bool
    }
}
```
`chatNote` 加**可选**参数（旧调用零改动）：`chatNote(messages:note:project: ProjectContext? = nil, kickoff:)`。

#### L3 摘要怎么取（不另造算法）
**直接复用 `NotesRepository.snippet(of:)`**（取第一个非空 paragraph/heading，截断），教练场景上限放宽到 **120 字**（约 60–90 token），前缀拼 title。
- 不用 fnv hash（指纹不可读、对 AI 无语义）；不做 AI 二次摘要（反向烧 token）。
- 跳过 🔒/pinned/module 块；现算即可（N≤8 篇、每篇扫到第一段就停），无需落盘缓存。

#### token 预算表（按「1 中文字≈1 token」保守估）
| 层 | 内容 | 软上限 | 硬上限 | 快取 |
| --- | --- | --- | --- | --- |
| L1 身份证 | SystemSpec | 800 | 1,000 | ephemeral 长 TTL(1h) |
| L2 当前笔记 | 全文(去 🔒/pinned/module) | 8,000 | 10,000 | ephemeral 短 TTL(5min) |
| L3 其他笔记摘要 | NoteDigest × N | 1,500 | 2,000 | 不快取 |
| 历史对话 | messages | 2,000 | 3,000 | 不快取 |
| 系统提示词 | 教练人格+模式 | 1,200 | 1,500 | 随 L1 挂长快取 |
| **输入合计** | | **~13.5k** | **17.5k** | |

#### 超预算裁剪顺序（在 App 端组好，别让后端裁）
铁律：**当前笔记 L2 永不裁**。由低到高裁：① L3 其他笔记（按 updatedAt 取最近 N=8 篇，逐篇累加超 2,000 就停、被截设 `truncated=true`）→ ② 历史对话（超 3,000 从最旧丢，保留 kickoff 首条）→ ③ L2 仅当单篇 >10k 才退回 diff/changed → ④ L1 不裁。

#### 落点（旧路径不动）
| 落点 | 改什么 |
| --- | --- |
| AIServicing.swift | 加 `ProjectContext`；`chatNote` 加 `project? = nil` |
| AIServiceLive.swift | chatNote Body 加可选 `project`，nil 不编码 |
| NoteDocument.swift | 加 `projectContext(repository:)`：L1 读 spec、L2 现有 payload、L3 调 `notes(in:)` 取前 8 篇做 digest |
| NotesRepository.swift | snippet 抽成 `digest(of:maxChars:)` 复用 |
| NoteAIViewModels.swift | runStream 加 `project?` 形参；**教练页走带 project 路径，单笔记页聊天传 nil** |

后端只需认 `project` 字段、L1/系统提示词标 cache_control、L3 不标快取。SSE 协议不动。

**⚠️ 依赖**：L1（SystemSpec）与 🔒（excludedFromAI）要等第 2/3 刀先落地；在那之前 spec 传 nil，教练只有 L2+L3。token 上限是保守默认值，待后端 `usage`/`count_tokens` 实测微调。
