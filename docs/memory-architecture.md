# BrainStrom · AI 记忆架构（待 MVP 验证）

> 状态：**已记录，待验证**。这是产品的地基，需要先做 MVP 把「LLM 记忆结构的边界」测清楚，再展开功能开发。
> 记录日期：2026-06-09

## 一句话

每一本笔记 = 一个「待开发的系统主题」。每个主题有**专属 AI 记忆**；其上还有一个**全局系统 AI** 掌握全部内容。

## 层级结构

```
┌─────────────────────────────────────────────┐
│  全局系统 AI（Global AI）                      │
│  · 掌握全部笔记本的内容                          │
│  · 负责跨主题检索、串联、回答                      │
│  · 解决「我记得有个灵感，但忘了写在哪本」           │
└───────────────┬─────────────────────────────┘
                │ 索引 / 路由
   ┌────────────┼────────────┬────────────┐
   ▼            ▼            ▼            ▼
┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
│主题AI A │ │主题AI B │ │主题AI C │ │  …     │
│只记得本 │ │只记得本 │ │只记得本 │ │        │
│A 的内容 │ │B 的内容 │ │C 的内容 │ │        │
└────────┘ └────────┘ └────────┘ └────────┘
 笔记本A     笔记本B     笔记本C
(系统主题)  (系统主题)  (系统主题)
```

- **第 1 层 · 主题专属 AI**：记忆被隔离/聚焦在单本笔记。回答这本内部的问题、帮这本补全/整理。
- **第 2 层 · 全局系统 AI**：跨所有笔记本。核心价值是**模糊回忆的跨本检索**——用户只记得灵感内容、不记得归属，全局 AI 帮他定位「这条原本在 X 主题里」。

## 为什么要先做 MVP（先测边界，再做功能）

要先验证「记忆结构」本身，搞清楚边界在哪：

1. **每个主题 AI 的记忆怎么存？** 全量塞进上下文窗口 vs 检索式（RAG / 向量库 + 摘要）？一本笔记长大后哪种撑得住？
2. **全局 AI 怎么跨本找？** 是给它所有笔记的索引/摘要，还是做向量检索后再交给它回答？跨本召回的准确率和边界在哪？
3. **两层之间怎么协作/路由？** 用户提问时，先问全局再下钻到主题，还是反过来？什么时候该用哪个？
4. **隔离 vs 共享的取舍**：主题 AI 的「聚焦」什么时候是优点（回答更准），什么时候是缺点（漏掉别处相关灵感）？
5. **模糊回忆这个核心场景**的成功率：给一句很模糊的描述，全局 AI 能不能稳定地找回正确那条 + 正确归属？

## 待验证的核心问题（MVP 要回答）

- [ ] 单本主题的记忆，用「全量上下文」能撑到多大？什么时候必须转检索式？
- [ ] 全局跨本检索（模糊回忆 → 定位归属）的准确率，用什么方案最稳？
- [ ] 两层 AI 的职责边界与路由策略
- [ ] 这套结构在「记忆会随使用不断增长」的真实场景下会不会崩
