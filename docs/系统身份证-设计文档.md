## 1. 一句话

每个「系统」配一张**固定格式的单页规格表＝系统身份证**，装着这个项目的「硬需求」（名称、技术栈、数据库、上线）；**选单一点就看**，取代散乱信息。**绝不强迫填**——靠 AI 在使用者写笔记/聊天时主动询问、一点即记，边做边长出来。

## 2. 为什么需要（设计哲学）

- 不管做什么系统，核心需求的结构大同小异，这就是「硬需求」。把它结构化集中，胜过散在笔记里。
- **开发顺序是模糊的**：你永远不知道使用者何时、如何变更计划。任何新功能、开发样式、后端需求都会改动身份证内容。
- **每个人背景不同**：有人从功能场景开始，有人一开始就清楚技术栈。所以设计必须**灵活、不强制、可随时增补**。

## 3. 字段（皆为选填，可扩充）

| 区块 | 字段 |
|---|---|
| 系统名称 | name |
| 技术栈 | 前端框架 frontend、后端框架 backend、API 使用 apis[] |
| 数据库 | database |
| 上线发布 | 服务器 server、发布方法 deployMethod |

- 全部可空；空着不影响使用。
- 预留扩充：金流平台、定价、目标用户、美学/样式库（后续视需要加，先不放）。
- 「美学/样式库」暂归「模组/笔记」区，不进身份证第一版。

## 4. 核心机制：AI 维护、使用者一点即记（绝不强迫）

复用**已实作的「AI 建议按钮（proposal）+ 点子助攻」系统**——身份证更新＝其延伸，不是从零做：

1. 使用者在聊天/写笔记时提到硬需求（如「我要用 React」「部署在 Fly.io」「资料库用 PostgreSQL」）。
2. AI 侦测到 → 在回答下方跳出建议钮：**「📇 记入身份证：前端 = React」**。
3. 使用者**点一下** → 该字段即填入身份证。
4. 计划变更（如资料库从 MySQL 改 PostgreSQL）→ AI 发现与已记内容不符 → 主动问「要更新吗？」。
5. **手动随时可改**：身份证单页随时打开、自己编辑，不必等 AI。
6. **AI 反向读取**：AI 帮忙（优化/结构化/聊天）时把身份证当上下文，知道「这系统用什么技术栈」，建议才准——这是身份证最大价值。

### 待拍板（讨论中倾向）
- 记入方式：用现成**聊天建议钮**（点了就填），不另弹窗打扰。【倾向采用】
- 建议粒度：一次可建议**一批**（前端/后端/部署一起），但使用者可个别取消。【倾向采用】
- 手动编辑：随时开启可改。【确定】

## 5. UI 呈现

- **入口**：筆記頁右缘「模组直条」最上方一颗 **🪪 身份证** 钮（也可于首页系统卡露出摘要），点开＝固定单页。
- **单页**：分区块（名称/技术栈/数据库/上线），每区一行行展示；空字段显示淡色占位「— 未填，可点编辑或让 AI 帮你记」；可点任一字段就地编辑。
- **样式**：沿用工业橘美学（铭板抬头、缺角卡、Tokens/Palette），与既有页面一致。

## 6. 资料模型与分工

- **资料**：在「系统」上加一个结构化 `SystemSpec`（上述字段，皆 optional），随系统持久化（SwiftData）。
- **前端（iOS 线）**：身份证单页（看＋手动改）、建议钮点下即填、AI 请求时把 spec 并入 note payload 当上下文。
- **后端（需配合）**：教 Claude 在侦测硬需求时主动丢出「update_spec」类建议（改 system prompt + 加一种 proposal action）。**此部分需后端线同意动 server/**。
- 结论：身份证的**资料、单页、手动编辑、AI 读取**纯前端可做；**AI 主动建议**需后端配合 prompt。

## 7. 与既有结构的关系

- 系统 = **一张身份证（固定结构化封面）** ＋ **一叠自由笔记（活文件：块/模组/开发日记）**。
- 身份证**不是塞进笔记的模组**，是系统层级的「封面/设定页」，与自由笔记并存。
